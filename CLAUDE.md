# claude.md

本文件定义本项目中所有 AI Agent、claude 及自动化开发工具必须遵守的长期开发规范。

优先级：

1. 用户当前明确指令
2. 本文件 `claude.md`
3. 项目施工规划与设计文档
4. 现有项目架构与代码规范

如不同规则存在冲突、含义不明确或无法确认，应暂停相关修改并向用户说明。

# 一、核心铁律

- 以瞎猜接口为耻，以认真查询为荣。
- 以模糊执行为耻，以寻求确认为荣。
- 以臆想业务为耻，以人类确认为荣。
- 以创造接口为耻，以复用现有为荣。
- 以跳过验证为耻，以主动测试为荣。
- 以破坏架构为耻，以遵循规范为荣。
- 以假装理解为耻，以诚实无知为荣。
- 以盲目修改为耻，以谨慎重构为荣。

# 二、工作原则

## 1. 先理解，再修改

修改代码前必须先确认：

- 当前需求
- 现有实现
- 调用链路
- 已有公共能力
- 可能影响的模块

禁止在未阅读相关代码的情况下直接重写。

## 2. 接口必须有依据

涉及以下内容时必须查询当前官方文档或现有项目实现：

- Discord API
- Discord Gateway
- Discord 权限
- Discord Events / Interactions
- DeepSeek API
- 第三方 SDK
- 外部服务接口

禁止凭记忆猜测：

- API 路径
- 事件名称
- 字段结构
- 权限要求
- 返回格式
- SDK 方法

如真实行为仍无法通过文档确认，应通过最小测试验证。

## 3. 优先复用

新增功能前先搜索项目中是否已有：

- API Client
- Discord Client
- AI Provider
- Logger
- Storage
- Message Sender
- Emoji Registry
- 配置读取
- 错误类型
- 通用工具

已有能力能够满足需求时应直接复用或合理扩展。

禁止为了局部方便重复制造相同能力。

## 4. 保持模块边界

项目长期按以下职责划分：

- `core`：启动、生命周期、事件调度
- `discord`：Discord 平台能力
- `ai`：AI Provider
- `features`：具体业务功能
- `storage`：持久化
- `resources`：表情等共享资源
- `config`：配置管理
- `utils`：真正通用的工具

Feature 负责业务规则。

公共基础模块不得反向依赖具体 Feature。

## 5. 避免过度设计

允许为已知扩展方向保留合理接口。

禁止因为“以后可能会用”提前开发：

- 复杂插件系统
- 多层抽象工厂
- 大型事件总线
- Web 管理后台
- 数据库框架
- 未确认的业务功能

优先选择当前足够、未来容易扩展的最小设计。

# 三、代码修改规范

## 1. 最小修改原则

修改已有功能时：

- 优先局部修复
- 保持现有行为兼容
- 避免无关格式化
- 避免顺手重构无关模块
- 避免扩大修改范围

发现额外问题时可以记录，但未经确认不要擅自扩大任务。

## 2. 禁止盲目兜底

错误必须尽量暴露真实原因。

禁止：

- 吞掉异常
- 空 `catch`
- 无日志降级
- 多层 fallback 掩盖根因
- 失败后假装成功
- 用默认值掩盖必要配置缺失

确实需要 fallback 时，必须有明确业务依据和日志。

## 3. 禁止散落硬编码

以下内容不得散落在业务代码：

- Token
- API Key
- Guild ID
- Channel ID
- Application Emoji ID
- API Base URL
- 模型名称
- 可配置业务参数

敏感信息进入 `.env`。

普通配置进入统一配置模块或资源配置。

## 4. 删除代码必须确认引用

删除任何：

- 函数
- 模块
- 配置项
- 兼容逻辑
- 公共接口

之前必须搜索全部引用并确认影响范围。

不得仅凭“看起来没用”判断为废代码。

# 四、测试与验证规范

每次修改必须执行与修改范围匹配的验证。

至少包括适用的：

- 静态检查
- 单元测试
- 模拟测试
- 启动测试
- API 调用测试
- Discord 真实环境测试

测试失败必须先分析原因，再决定修改。

不得通过删除测试、放宽断言或跳过检查制造“通过”。

## 外部环境验证

以下内容无法仅靠模拟测试最终确认：

- Discord Bot 实际登录
- Discord 权限
- 真实 Boost 事件
- Application Emoji Reaction
- 指定频道发送行为

此类功能可以完成代码和自动化测试，但最终状态必须明确标记为：

`待真实环境验证`

不得在未经真实测试的情况下宣称完全验收通过。

# 五、错误处理规范

错误应在最接近问题来源的合理层级处理。

日志必须能够回答：

- 哪里失败
- 什么操作失败
- 错误原因是什么
- 是否会重试
- 是否影响核心业务

不得记录：

- Discord Bot Token
- DeepSeek API Key
- 其他完整敏感凭证

# 六、业务开发规范

当前每个业务功能应作为独立 Feature。

例如：

features/
├── boostThanks/
├── announcements/
├── channelSummary/
└── chat/

新增 Feature 应尽量复用公共能力。

新增一个 Feature 原则上不应要求重写已有 Feature。

# 七、AI 使用规范

AI Provider 负责：

- 请求模型
- 参数处理
- 超时
- API 错误
- 返回结果标准化

Feature 负责：

- Prompt 内容
- 业务上下文
- 结果校验
- 后续业务流程

禁止 Feature 自行重复实现 DeepSeek API Client。

禁止让 AI 生成或猜测：

- Discord Emoji ID
- Channel ID
- Guild ID
- API 字段
- 权限配置

这些信息必须来自真实配置或已验证数据。

# 八、Git 与安全规范

必须确保以下内容不会提交：

- `.env`
- Token
- API Key
- 私密配置
- 调试产生的敏感数据

提供 `.env.example` 时只能包含变量名和安全示例。

提交前检查：

- `git diff`
- 新增文件
- 敏感信息
- 意外生成文件

# 九、任务执行规则

开始任务前：

1. 阅读本文件。
2. 阅读当前施工规划。
3. 检查现有项目结构。
4. 搜索相关实现。
5. 确认修改范围。

执行过程中：

1. 小步修改。
2. 及时验证。
3. 发现问题定位根因。
4. 不扩大未经确认的需求。

完成任务后必须汇报：

- 修改内容
- 修改文件
- 测试内容
- 测试结果
- 未验证事项
- 已知风险

如遇无法确认的问题，应明确说明未知点并暂停相关决策。

# 十、最终原则

代码应追求：

- 清晰
- 简单
- 可验证
- 可维护
- 可扩展

扩展性来自清晰的模块边界与稳定的公共能力。

不要通过提前堆积复杂度换取想象中的未来便利。
